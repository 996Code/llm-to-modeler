# 意图识别节点实现计划

## 核心改动

在管线最前面加 `classify_intent` 节点（LLM），识别 3 种意图后路由到不同管线。前端统一调 `/api/chat`，不再区分 generate/modify。

## 意图定义

| 意图 | 含义 | 示例 |
|------|------|------|
| `create` | 创建新表单 | "创建一个请假表"、"新建一个客户信息表" |
| `modify` | 修改已有表单 | "加一个手机号字段"、"把姓名改成必填"、"删除这个字段" |
| `general` | 闲聊/打招呼 | "你好"、"谢谢"、"你是谁" |

> CLARIFICATION（追问）仍然保留在 `parse_fields` 节点内部，不走意图识别——意图只判断"做什么类型的事"，parse_fields 判断"需求够不够详细"。

## 管线结构（改动后）

```
START → classify_intent (LLM)
  ├─ create  → fetch_guide → list_assets → parse_fields → fetch_templates → generate → validate → END
  │                                              ↓ clarify
  │                                            END
  ├─ modify  → fetch_guide → modify → validate → (retry?modify:END)
  └─ general → general_reply → END
```

关键：`classify_intent` 和 `fetch_guide` 之间用条件边，`fetch_guide` 之后再用条件边根据 `state.intent` 分流到 CREATE 或 MODIFY 管线。

## 文件改动清单

### 后端（6 个文件）

**1. `state.py`** — 新增字段
```python
intent: str = ""              # "create" | "modify" | "general"
intent_reason: str = ""       # LLM 给出的判断理由
general_reply: str = ""       # general 意图的回复文本
```

**2. `nodes.py`** — 新增 2 个节点 + 2 个条件边函数
- `classify_intent_node` — 调 LLM 判断意图，注入历史上下文（对标 chat-bi 的 classify_intent）
- `general_reply_node` — 闲聊回复（不走表单管线）
- `route_by_intent(state)` → `"fetch_guide" | "general_reply"`
- `route_after_guide(state)` → `"list_assets" | "modify"`

**3. `prompt_builder.py`** — 新增 `build_intent_prompt()`
- 3 种意图定义 + 判断规则
- 注入 `compressed_history` 和 `has_existing_config` 标记到 user message
- 输出 JSON: `{intent, reason}`

**4. `graph.py`** — 统一管线（移除 `is_modify` 参数）
- `_build_graph()` 不再接收 `is_modify`，构建一张完整的图
- 两条条件边：classify_intent → {fetch_guide, general_reply}；fetch_guide → {list_assets, modify}
- `run()` 不再判断 is_modify，统一入口

**5. `api/config.py`** — 新增 `POST /api/chat` 统一入口
- 请求体: `{message, conversation_id}`
- 从 conversation store 加载 history + current_config
- 传 `current_config` 给 workflow（后端自动判断意图，不再依赖前端指定）
- 保留 `/generate` 和 `/modify` 向后兼容

**6. `api/sse.py`** — 处理 general_reply 结果
- `final_state.intent == "general"` 时返回 `{intent: "general", reply: ..., summary: ...}`
- 保存 general 回复到对话历史

### 前端（4 个文件）

**7. `services/api.ts`** — 新增 `chat()` 函数
- `POST /api/chat {message, conversation_id}`，SSE 回调同现有

**8. `stores/conversation.ts`** — 简化
- `sendMessage` 改为调 `api.chat()`（不再区分 generate/modify）
- 移除 `sendModify`（或保留但内部也调 chat）
- `onResult` 处理 3 种结果：config / needsClarification / general reply
- 根据 SSE stage 事件动态设置 `streamingModify`（后端 emit intent 后前端知道走哪条管线）

**9. `components/chat/ChatPanel.vue`** — 更新管线步骤
- 新增 intent 步骤（"理解意图"）
- 根据 SSE 返回的 intent 动态切换 CREATE_STAGES / MODIFY_STAGES
- 处理 general reply（显示普通文本，不显示配置卡片）
- `handleSend` 不再判断 hasConfig

**10. `components/chat/ChatInput.vue`** — 移除 hasConfig 分支
- placeholder 不再根据 hasConfig 变化（统一为"描述你的需求..."）

## 端到端验证

1. "你好" → general → 闲聊回复（不走管线）
2. "创建一个请假申请表，包含申请人、请假类型、开始日期、结束日期" → create → 6 步生成
3. "加一个请假原因字段" → modify → 3 步修改（保留原有字段）
4. "创建一个表单" → create → parse_fields 触发追问
5. "把姓名改成必填" → modify → 修改配置
